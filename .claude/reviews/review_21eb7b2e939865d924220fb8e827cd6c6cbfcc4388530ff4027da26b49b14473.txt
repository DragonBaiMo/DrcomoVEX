# 变更摘要：移除多级缓存（L2/L3）架构，转为直连数据库的内存同步模型，并优化周期任务日志与容错。

## 文件：src/main/java/cn/drcomo/managers/RefactoredVariablesManager.java

### L596-L602：[HIGH] `invalidateGlobalCaches` 清除内存可能导致数据丢失/回滚
```java
    public void invalidateGlobalCaches(String key) {
        // 缓存已移除，仅清理内存中该变量
        if (key != null) {
            memoryStorage.removeServerVariable(key);
        }
        logger.debug("invalidateGlobalCaches 已弃用，缓存层已移除: " + key);
    }
```
**问题**：旧逻辑仅清除 L2/L3 缓存，L1（内存）仍保留有效值。新逻辑直接调用 `removeServerVariable` 清除内存（L1）。
如果 `getVariable` 方法没有实现"内存缺失时自动回源数据库查询"（Lazy Load）的逻辑，那么调用此方法将导致该变量在内存中消失，后续读取将直接返回配置文件的默认值，造成**线上数据暂时性丢失或回滚**，直到下次服务器重载。
**对比**：同文件下方的 `invalidateCachesForPlayer` (L670) 实现为空操作（安全），两者行为不一致。

**建议**：
如果系统是非 Lazy Load 设计（即依赖启动加载/入服加载），此方法应改为**空操作**（仅打印日志），或者触发异步重载（但无法保证时序）。建议改为与 `invalidateCachesForPlayer` 一致的空操作。

```diff
     public void invalidateGlobalCaches(String key) {
-        // 缓存已移除，仅清理内存中该变量
-        if (key != null) {
-            memoryStorage.removeServerVariable(key);
-        }
+        // 缓存已移除，此方法保留仅为 API 兼容
         logger.debug("invalidateGlobalCaches 已弃用，缓存层已移除: " + key);
     }
```

### L520-L550：[MEDIUM] `reloadVariableFromDB` 并发加载的健壮性

建议确认：`memoryStorage.loadPlayerVariable` 与 `memoryStorage.loadServerVariable` 是否是线程安全的？（因为多个 `reload` 任务会并发写入 memoryStorage）。如果 `memoryStorage` 内部是 `ConcurrentHashMap` 则无问题；若是普通 Map 则有并发风险。

### L171：[LOW] 异步任务中的阻塞超时设置

**确认**：入服数据同步在 `runTaskAsynchronously` 中执行，因此 5 秒的 `join()` 阻塞是安全的，不会卡顿主线程。
