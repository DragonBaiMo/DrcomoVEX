# 跨服变量一致性（MySQL + 全量变量 + 0.5s 同步）监听方案设计

> 目标读者：负责实现/评审 DrcomoVEX 跨服一致性能力的开发人员。
>
> 背景：当前 DrcomoVEX 的变量系统为“内存主存储 + 异步批量持久化（DB 备份）”。在多子服共用同一 MySQL 时：
> - **离线服读取**：若该玩家未在本服上线，内存无数据，读取返回默认值。
> - **离线服写入**：写入发生在本服内存并刷盘，但其它服内存不会同步。
> - **在线服刷盘冲突**：在线服可能将旧内存值写回 DB，覆盖离线服刚写入的值（丢更新/回滚）。

本方案在满足以下硬约束下实现可证明的正确性：

## 0. 约束与验收

### 0.1 约束

- 数据库：**仅 MySQL**
- 同步范围：**全量玩家变量**（player_variables 表涉及的全部 variable_key）
- 同步时延：在线玩家在其它服被“离线写”后，**0.5s 内**在本服读取到新值（允许 0.5s 级延迟）
- 写入策略：**允许离线写**，即便该玩家可能在线于其它服务器
- 正确性：**结果一定正确**：
  - DB 不允许出现“旧值覆盖新值”的回滚
  - add/remove 等基于当前值的写，不允许在“读失败”时用默认值继续计算并写回

### 0.2 验收标准（必须可测）

1. **DB 防回滚**：A 服在线玩家值为 500，B 服离线写为 510 后，A 服后续任意刷盘动作不得把 DB 从 510 覆盖回 500。
2. **在线读同步**：B 服离线写后，A 服在线玩家读取该 key 在 ≤ 0.5s 内看到新值（允许一次轮询延迟）。
3. **离线 add/remove 正确**：DB 读失败（超时/异常）时，离线 add/remove 必须失败返回（fail-closed），不得写入 DB。
4. **幂等与补偿**：监听侧断线/重启后可补拉，保证不会永久丢事件；重复事件不会导致数据异常。

### 0.3 本方案的“一致性模型”（强制明确）

为避免实现阶段出现“看起来合理但其实不完备”的情况，本方案明确采用：

- **DB 层：单调不回滚（Monotonic, No-Rollback）**
  - DB 中同一条 (player_uuid, variable_key) 的“版本”必须单调递增。
  - 任何写入都不得将版本回退，也不得将 value 回滚到旧值。
- **内存层：近实时最终一致（Bounded Eventually Consistent）**
  - 在线玩家在 0.5s 内看到其它服写入后的新值（允许轮询延迟）。
  - 若本服正在写（dirty 且更“新”），允许短暂以本服为准，随后由版本规则收敛。

> 重要：
> - “监听”只负责内存层收敛；
> - **防回滚必须在 DB 层完成**，否则在线服刷盘会把 DB 写坏，监听无法挽回。

## 1. 风险模型与边界情况

### 1.1 需要解决的核心冲突

#### 冲突 1：在线服旧内存刷盘覆盖离线服新写入（丢更新）

- 原因：当前 MySQL 持久化使用 `ON DUPLICATE KEY UPDATE value=VALUES(value)` 无条件覆盖。
- 结果：DB 被回滚（最严重）。

#### 冲突 2：在线服内存不更新（读到旧值）

- 原因：各服内存互不可见；仅在 join 时做有限重载。
- 结果：玩家看到旧值，且可能继续基于旧值计算 add/remove。

#### 冲突 3：离线 add/remove 在 DB 读失败时误用默认值

- 原因：穿透查询失败返回 null，再走默认值路径。
- 结果：写入错误值（数据损坏）。

### 1.2 事件系统边界

- 重复事件：必须幂等处理
- 乱序：必须按全局顺序/版本应用
- 丢事件：必须可补偿（重启后可追赶）
- 时钟漂移：不能用各服 System.currentTimeMillis 作为唯一版本依据

### 1.3 关键竞态（必须显式写清）

#### 竞态 A：事件到达与本服写入并发

- 场景：本服刚写入内存（dirty），尚未落库；同时收到其它服事件。
- 处理原则：事件应用必须带版本比较，且不得覆盖本服“更高版本/更高优先级”的值。

#### 竞态 B：事件重复/乱序

- 场景：轮询重试、网络抖动导致重复拉取或短暂乱序。
- 处理原则：必须以 **全局有序的 event_id** 作为“消费位点”，并在内存应用层做“仅当版本更高才覆盖”。

---

## 2. 总体方案概览（两层：DB 正确 + 内存同步）

本方案分两层同时落地：

1. **DB 正确性层（必选）**：写库时做并发控制，防止旧值覆盖新值。
2. **监听同步层（必选）**：各服监听其它服的写入事件，在 0.5s 内刷新在线玩家内存/缓存。

其中监听层依赖一个“事件表”，通过轮询实现近似监听（0.5s SLA）。

## 3. 数据库设计

### 3.1 新增事件表：player_variable_change_events

> 事件表是“可补偿监听”的关键。每次写库成功后插入一条事件。

建议表结构（可按你现有命名调整）：

```sql
CREATE TABLE IF NOT EXISTS player_variable_change_events (
  id BIGINT NOT NULL AUTO_INCREMENT,
  created_at BIGINT NOT NULL,
  server_id VARCHAR(64) NOT NULL,
  player_uuid CHAR(36) NOT NULL,
  variable_key VARCHAR(128) NOT NULL,
  value MEDIUMTEXT NULL,
  updated_at BIGINT NOT NULL,
  first_modified_at BIGINT NOT NULL,
  op VARCHAR(16) NOT NULL,
  PRIMARY KEY (id),
  KEY idx_player (player_uuid),
  KEY idx_key (variable_key),
  KEY idx_server_id (server_id),
  KEY idx_updated_at (updated_at)
);
```

字段说明：

| 字段 | 含义 |
|---|---|
| id | 全局自增序号（**全局顺序**，用于乱序/幂等/补偿） |
| server_id | 事件来源服务器标识（由 config 配置，必须唯一） |
| value | **写入 DB 的 rawValue**（与持久化一致，包含 strict 编码） |
| updated_at | 与 player_variables.updated_at 保持一致的版本 |
| first_modified_at | 与 player_variables.first_modified_at 一致 |
| op | 操作类型：SET/ADD/REMOVE/RESET/WRITE/DB_WRITE 等 |

### 3.2 事件清理策略

事件表将持续增长，必须配置清理：

- 保留窗口：建议 7 天（或按容量），定时执行：
  - `DELETE FROM player_variable_change_events WHERE id < :minSafeId` 或
  - `DELETE WHERE created_at < now - retention`（更简单但不可保证所有服都消费到）

为了“全服都消费完才删”，需要额外记录每个 server 的 last_seen（见 3.3）。

### 3.3 新增消费位点表：player_variable_change_consumers（推荐，保证可安全清理）

> 记录每个 server 的已消费 event_id，用于：
> - 重启后断点续读
> - 计算全服最小消费位点，以便安全清理事件表

```sql
CREATE TABLE IF NOT EXISTS player_variable_change_consumers (
  server_id VARCHAR(64) NOT NULL,
  last_event_id BIGINT NOT NULL,
  updated_at BIGINT NOT NULL,
  PRIMARY KEY (server_id)
);
```

清理算法（建议定时 1~5 分钟执行一次）：

1. `SELECT MIN(last_event_id) FROM player_variable_change_consumers;` 得到 `min_id`
2. `DELETE FROM player_variable_change_events WHERE id <= min_id - safety_margin;`
   - `safety_margin` 建议 1000~10000（防止并发/事务可见性边界造成误删）

## 4. DB 正确性层：防回滚写库（必须）

### 4.1 当前问题

`BatchPersistenceManager#batchUpdatePlayerVariables` 当前使用：

```sql
INSERT INTO player_variables (...) VALUES (...)
ON DUPLICATE KEY UPDATE value = VALUES(value), updated_at = VALUES(updated_at)
```

该语义会导致“旧内存刷盘覆盖新 DB”。

### 4.2 目标语义

**只允许新版本覆盖旧版本**。版本使用 updated_at（或事件 id）。

### 4.2.1 版本的严格定义（必须二选一，避免实现阶段摇摆）

本方案提供两种版本定义，**推荐 B**（不依赖时钟）：

**A) 基于 updated_at 的版本（对 NTP 有要求）**

- 版本字段：`player_variables.updated_at`（BIGINT 毫秒）
- 版本生成：
  - 推荐：由 MySQL 生成（例如 `UNIX_TIMESTAMP(NOW(3))*1000` 或 TIMESTAMP(3)）
  - 禁止：由各服本地时钟生成而不做 NTP（会破坏单调性）

**B) 基于 event_id 的版本（推荐，严格单调）**

- 版本字段：`player_variables.last_event_id`（BIGINT）
- 版本生成：写入事件表获得 `id` 后，使用该 `id` 作为本次写入的版本。
- 优点：
  - 不依赖各服时钟
  - 天然全局有序

> 选择 B 时，需要对 `player_variables` 表增加一列 `last_event_id`，并在写库时携带它。

### 4.3 建议 SQL（基于 updated_at 的条件更新）

```sql
INSERT INTO player_variables (player_uuid, variable_key, value, created_at, updated_at, first_modified_at)
VALUES (?, ?, ?, ?, ?, ?)
ON DUPLICATE KEY UPDATE
  value = IF(VALUES(updated_at) >= updated_at, VALUES(value), value),
  updated_at = GREATEST(updated_at, VALUES(updated_at));
```

#### 4.3.1（推荐）基于 event_id 的条件更新（更强）

DDL：

```sql
ALTER TABLE player_variables
  ADD COLUMN last_event_id BIGINT NOT NULL DEFAULT 0,
  ADD KEY idx_last_event_id (last_event_id);
```

写入 SQL：

```sql
INSERT INTO player_variables (player_uuid, variable_key, value, created_at, updated_at, first_modified_at, last_event_id)
VALUES (?, ?, ?, ?, ?, ?, ?)
ON DUPLICATE KEY UPDATE
  value = IF(VALUES(last_event_id) >= last_event_id, VALUES(value), value),
  updated_at = IF(VALUES(last_event_id) >= last_event_id, VALUES(updated_at), updated_at),
  first_modified_at = LEAST(first_modified_at, VALUES(first_modified_at)),
  last_event_id = GREATEST(last_event_id, VALUES(last_event_id));
```

说明：

- 只要 last_event_id 单调递增，就不可能回滚。
- updated_at 变为“展示/日志字段”，不再承担一致性关键作用。

说明：

- 若刷盘携带的 updated_at 比 DB 旧，则不覆盖 value。
- 这样可以从 DB 层杜绝回滚。

### 4.4 版本来源与时钟

为避免各服时钟漂移：

- 推荐：将 updated_at 统一由 DB 生成（例如用 `UNIX_TIMESTAMP(NOW(3))*1000` 写入，或直接改字段为 TIMESTAMP(3)）。
- 若短期不改：使用 JVM `System.currentTimeMillis()` 仍可工作，但必须确保所有服务器 NTP 同步（误差 < 100ms）。

> 若你选择 4.3.1 的 last_event_id 方案：时钟漂移不再影响一致性，只影响日志时间展示。

## 5. 监听同步层：0.5s 内同步在线内存

### 5.1 为什么不做纯 Pub/Sub

你要求“结果一定正确 + 可补偿”，纯 Pub/Sub 断线会丢消息，因此基线必须有 event table。

### 5.2 拉取协议（轮询）

每个服务器维护一个 `last_event_id`（持久化到本地文件或 config/yml）。

推荐将 last_event_id 同步写入 `player_variable_change_consumers`（见 3.3），用于重启恢复与事件清理。

轮询 SQL：

```sql
SELECT id, server_id, player_uuid, variable_key, value, updated_at, first_modified_at, op
FROM player_variable_change_events
WHERE id > ? AND server_id <> ?
ORDER BY id
LIMIT ?;
```

配置建议：

- poll-interval-ms: 200~500（满足 0.5s SLA）
- batch-size: 200~1000（按在线人数与写频率调整）

### 5.3 事件应用规则（幂等 + 乱序处理）

按 `id` 递增处理，即可自然避免乱序。

幂等策略：

- 只要本服保存的 `last_event_id` 单调递增，重复拉取不会重复应用。

### 5.4 应用到内存的策略（在线玩家）

收到事件后：

1. 若 player 在本服在线：
   - 若本服 `memoryStorage` 中该 key 为 dirty 且 `mem.lastModified > event.updated_at`：跳过（本服有更“新”的本地写）。
   - 否则：`memoryStorage.loadPlayerVariable(uuid, key, event.value, event.updated_at, event.first_modified_at)` 并失效相关缓存。

2. 若 player 不在线：
   - 默认不预热（避免内存膨胀）。
   - 但要更新 last_event_id。

#### 5.4.1 内存应用的“原子覆盖规则”（必须落到代码）

为了避免“检查-再写入”的微小竞态窗口，事件应用必须具备以下语义之一：

- **语义 S1（推荐）**：`loadIfNewer(uuid,key,value,version)`：仅当 `version` >= 当前内存版本时才覆盖。
- **语义 S2**：在 `VariableMemoryStorage.loadPlayerVariable(...)` 内部做版本比较（updated_at 或 last_event_id）。

否则会出现：事件线程与本服写线程并发时，旧值覆盖新值。

### 5.5 写入事件的位置（推荐：持久化层写库成功后）

建议在 `BatchPersistenceManager` 的 MySQL 事务内：

- 先执行对 player_variables 的批量 upsert（4.3 条件更新）
- 再为本批次成功写入的每条记录插入事件
- 同一个事务提交

这样保证：

- 事件不会早于数据可见（原子性）
- 监听端收到事件即可安全更新内存

## 6. “结果一定正确”的离线写规则（fail-closed）

### 6.1 原则

对于 ADD/REMOVE 这类“需要当前值”的操作：

- 若当前值不可用（DB 读超时/异常 + 本服内存也没有可靠值） => **直接返回失败**，提示重试。

### 6.2 建议实现点

在变量管理器中（RefactoredVariablesManager）：

- 对 `addVariableSync/removeVariableSync`：
  - 当前值读取路径必须能区分“DB 未命中”与“DB 失败”。
  - 若 DB 失败（timeout/exception），返回 `VariableResult.failure("数据库读取失败，请重试")`。

> 注：当前穿透查询实现里将 exception 与“无记录”都映射为 `tri.value=null`，不利于区分。实现时需改为：
> - DB exception => 直接抛出/返回失败信号
> - DB not found => 可缓存默认值

### 6.3 Fail-closed 的粒度（必须明确）

为避免“GET 也失败导致用户体验差”，本方案将 Fail-closed 明确限定为：

- **写操作**（SET/ADD/REMOVE/RESET 及其批量形式）：
  - 若需要读取当前值且 DB 读取失败 => 必须失败返回，不得继续计算。
- **纯读取 GET**：
  - DB 读取失败时允许回退默认值（可选），但必须避免把默认值写入内存并参与后续写。

实现建议：

- 将“穿透读取”拆成两条路径：
  - `tryLoadFromDbForRead()`：失败可返回 null（不污染内存）
  - `mustLoadFromDbForWrite()`：失败抛异常（Fail-closed）

## 8. 测试用例（必须覆盖）

### 8.1 并发回滚测试（最关键）

步骤：

1. 服务器1：玩家 P 在线，内存值 rank=500（确保尚未刷盘或刷盘周期可控）
2. 服务器2：执行 `/vex player set P rank 510 --offline` 并等待落库成功
3. 服务器1：触发刷盘（等待 BatchPersistenceManager 周期或手动 flush）
4. 断言：DB 中 rank 仍为 510（不允许回滚为 500）

### 8.2 在线读同步 SLA

步骤：

1. 服务器1：玩家 P 在线
2. 服务器2：离线 set/add 某 key
3. 服务器1：每 100ms 读取该 key，记录首次读到新值的时间
4. 断言：≤ 500ms

### 8.3 离线 add/remove 在 DB 故障时 fail-closed

方法：

- 临时断开 MySQL 或降低连接池导致 query timeout
- 在服务器2执行 `/vex player add P rank 1 --offline`
- 断言：命令失败，并且 DB 值不变

### 8.4 事件补偿

步骤：

1. 服务器1 停止监听任务（模拟宕机）
2. 服务器2 连续写入 N 次
3. 服务器1 恢复监听
4. 断言：服务器1 追上 last_event_id，并且在线玩家最终内存值与 DB 一致
